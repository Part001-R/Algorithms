### Authorization: Bearer <токен>
При передаче токена в заголовке Authorization использование слова Bearer указывает на тип токена, который передается. Это означает, что токен уже был выдан сервером при аутентификации пользователя.

Получение токена: Клиент, например пользовательское приложение или веб-сайт, проходит аутентификацию на сервере (например, с помощью логина и пароля). После успешной аутентификации сервер выдает токен доступа (Bearer token).
Клиентский запрос: Когда клиент хочет получить доступ к защищенному ресурс, он отправляет запрос к серверу, добавляя заголовок Authorization.

Здесь <токен> заменяется на токен, полученный ранее.
Проверка на сервере: Сервер извлекает токен из заголовка, проверяет его валидность, а также истечение срока действия. Если токен действителен, сервер предоставляет запрашиваемые ресурсы.

### Authorization: Basic <закодированная строка>
Basic: Указывает на использование базовой аутентификации, где в заголовке передается закодированная в Base64 пара «логин:пароль».

Запрос на доступ: Когда клиент пытается получить доступ к защищенному ресурсу, сервер отвечает сообщением 401 Unauthorized. В ответе содержится заголовок WWW-Authenticate, указывающий, что требуется аутентификация.
Клиентский запрос: Клиент отправляет новый запрос, включая заголовок Authorization, который содержит закодированную в Base64 строку. 

<закодированная строка> — это закодированный в Base64 результат комбинации «логин:пароль».

Формирование закодированной строки:
Например, для логина user и пароля pass создаётся строка user:pass, которая затем кодируется в Base64. В результате получится нечто вроде dXNlcjpwYXNz.
Проверка на сервере: Сервер декодирует полученный заголовок и сверяет логин и пароль с сохраненными данными. Если данные совпадают, доступ предоставляется.

### Authorization: Digest <данные>
Digest: Используется для более сложной аутентификации, которая обеспечивает дополнительные меры безопасности, такие как использование хеширования.

Запрос на аутентификацию: Когда клиент обращается к защищённому ресурсу, сервер отвечает сообщением 401 Unauthorized с заголовком WWW-Authenticate, указывая, что требуется аутентификация по схеме Digest.
Клиентский запрос: Клиент делает новый запрос с заголовком Authorization, который включает несколько полей:
- username: имя пользователя.
- realm: область, которая требует аутентификации.
- nonce: случайное значение, выданное сервером, чтобы предотвратить повторное использование токена.
- uri: путь к запрашиваемому ресурсу.
- response: хеш, который вычисляется на основе всех предыдущих значений, включая пароль пользователя, что делает его сложнее для подделки.

Хеширование: Для вычисления response используется следующий алгоритм:
- Сначала создаётся хешируемая строка с использованием метода MD5 (или другого, если указано).
- Хеш вычисляется на основе данных, включая полученные от сервера nonce, username, password и uri.
Проверка на сервере: Сервер вычисляет свой собственный response с использованием тех же данных и сверяет его с тем, что отправил клиент. Если они совпадают, доступ предоставляется.

### Authorization: ApiKey <ключ>
Authorization: ApiKey <ключ>

Метод проверки подлинности, при котором клиент предоставляет уникальный ключ для доступа к API или защищённым ресурсам. Этот подход часто используется для управления доступом к веб-сервисам и API.

Получение ключа: Клиент регистрируется на платформе или сервисе, чтобы получить уникальный API ключ. Этот ключ обычно представляет собой длинную строку, которая идентифицирует клиентское приложение.
Клиентский запрос: Когда клиент хочет получить доступ к защищённому ресурсу, он включает API ключ в заголовок запроса.

В этом формате <ключ> заменяется на фактический API ключ, полученный при регистрации.
Проверка на сервере: Сервер принимает ключ и проверяет его на действительность. Если ключ валиден и у него есть разрешения на доступ к запрашиваемым ресурсам, сервер разрешает доступ.
